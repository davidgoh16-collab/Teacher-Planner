# Teacher Planner

A multi-tenant teacher planner PWA: two-week timetable, lesson planning, projects & tasks,
routines, key dates, a meeting planner (compare staff/student timetables to find free slots),
AI-assisted imports and an AI assistant — built with React 19 + Vite + Tailwind (CDN),
Firebase (Auth + Firestore) and the Gemini API (`@google/genai`).

## Run locally

**Prerequisites:** Node.js 20+

1. Install dependencies:

   ```bash
   npm install
   ```

2. Create `.env.local` with your Gemini API key (get one at https://aistudio.google.com/apikey):

   ```bash
   VITE_GEMINI_API_KEY=your-key-here
   ```

   Every AI feature (timetable/term imports, chat assistant, insights, voice assistant) needs
   this key. Note that in a client-only app the key ships to the browser — restrict it to the
   Generative Language API in Google Cloud console.

3. Run the app:

   ```bash
   npm run dev
   ```

`npm run lint` type-checks, `npm run build` produces the production bundle (with the PWA
service worker generated by vite-plugin-pwa).

## Firebase setup (required for sharing)

The Firestore **security rules in `firestore.rules` must be deployed** or sharing will fail
with permission errors (stuck at "user lookup blocked" / "sharing isn't set up on the server"):

```bash
npx firebase login          # once
npm run rules:deploy        # deploys firestore.rules to the project in .firebaserc
```

Test the rules locally (starts the Firestore emulator, requires Java):

```bash
npm run rules:test
```

Auth providers (Google, Microsoft, Email/Password) are enabled in the Firebase console. The
Firebase web API key can be overridden with `VITE_FIREBASE_API_KEY` (it is an identifier, not
a secret — access control lives entirely in the rules).

## Deployment (Docker)

The `Dockerfile` builds the Vite bundle and serves it with nginx. `start-nginx.sh` writes
`env.js` at container start from the runtime environment, so set these on the container:

- `GEMINI_API_KEY` (or `VITE_GEMINI_API_KEY`) — Gemini API key
- `VITE_FIREBASE_API_KEY` — optional Firebase web key override
- `PORT` — port for nginx (e.g. Cloud Run sets this)

## Imports — what each one accepts

| Import | Where | Accepted input |
|---|---|---|
| Your timetable | Onboarding · Settings → Timetables | Photo/screenshot (PNG, JPG), PDF, Word (.docx), Excel (.xlsx/.xls/.ods), CSV/TSV, plain text, or paste text |
| Timetable + term dates together | Settings → Timetables → Auto-Import | Same as above |
| Term dates from your school website | Onboarding · Settings → Terms | Link to a **public** webpage (no login) listing term dates |
| Staff / student timetables (batch) | Meeting Planner → Import · Onboarding | Several files at once — one person per file; names are read from the document, falling back to the file name (name files like `J Smith.pdf`) |
| Assistant file attachments | Chat · Project assistant | PDF, images, Word, Excel/CSV, PowerPoint (.pptx), text/markdown/JSON |

Tips: a straight-on screenshot of the full grid (with day and period headers) parses best;
PDF exports from SIMS/Bromcom/Arbor work well; when pasting, copy the cells straight from
Excel/your MIS including the headers. Everything imported lands in a review step before it
is saved.

## Sharing

- Share your **timetable** (live, read-only) or a **project** (view or edit) from the share
  buttons; recipients see them under **Shared**.
- Shares are matched by the **email address the recipient signs in with**.
- If the person hasn't signed up yet, the share is saved as a **pending invite** and activates
  automatically the first time they sign in with that email. The dialog offers a prefilled
  **email draft** / copy-to-clipboard invite so you can let them know — the app does not send
  emails itself.
- Note: whoever first signs in with an email address receives the shares addressed to it, and
  email/password sign-ups are not verified. Don't share sensitive data with an address you
  don't trust.

## Development notes

- `?bypass_login=true` skips auth **in dev builds only** (for Playwright/UI smoke tests); it
  is compiled out of production bundles.
- Firestore data is scoped per user under `users/{uid}/…`; `shares/{id}` docs double as the
  sharing registry and the access grants (see `firestore.rules`).
- Rules tests live in `tests/rules/` and run against the Firestore emulator.
